## Context

Bullpen 是一个基于 SwiftUI / iOS 17+ / Tuist 的新项目，目前处于 Hello World 阶段（BullpenApp.swift + ContentView.swift）。需要从零构建守备练场功能，没有现有架构约束。

项目使用 Tuist 管理，源码在 `Bullpen/Sources/`，资源在 `Bullpen/Resources/`。

## Goals / Non-Goals

**Goals:**
- 构建可独立运行的守备练场功能，包含完整的场面生成 → 用户判断 → 判定反馈循环
- 规则引擎基于棒球规则动态推导，而非硬编码答案表
- 半图形化 UI，以棒球钻石展示场面，直观清晰
- 代码模块化，模型层与 UI 层清晰分离

**Non-Goals:**
- 来球方向/速度模拟
- 难度等级系统
- 双步操作（如双杀的两步传球）
- 外野传切配合（cutoff relay）
- 跑者速度差异
- 数据持久化（历史记录、成就等）
- 网络功能

## Decisions

### 1. 架构模式：MVVM

**选择**: 使用 SwiftUI 原生的 MVVM 模式，ViewModel 持有游戏状态和业务逻辑。

**理由**: 项目处于早期阶段，MVVM 与 SwiftUI 天然契合，复杂度适中。TCA 等架构对当前规模来说过重。

**替代方案**: TCA (The Composable Architecture) — 状态管理更严格，但引入外部依赖且学习成本高，当前功能规模不需要。

### 2. 规则引擎：纯 Swift 函数式实现

**选择**: 规则引擎作为无状态纯函数实现，输入 `(Position, Situation)` → 输出 `Answer`。

**理由**:
- 纯函数便于单元测试，每个场景都可以独立验证
- 无状态设计让规则引擎可以独立于 UI 测试
- Swift 的 enum + pattern matching 天然适合表达规则分支

**替代方案**: 数据驱动规则表（JSON/plist）— 灵活但调试困难，且棒球规则的条件分支用数据结构表达反而更复杂。

### 3. 答案判定：双层结构 (best / acceptable)

**选择**: Answer 包含 `best: [Action]` 和 `acceptable: [Action]` 两个层级，加上 `explanation: String`。

**理由**: 棒球守备很多场景有多个合理选择，双层结构既能给出最佳指导，又不会错判合理选择为"错误"，具有教学价值。

### 4. 场面生成：随机 + 位置权重过滤

**选择**: 随机生成场面组合，但按位置过滤不合理的来球类型（如外野手不出现短打）。

**理由**: MVP 不做难度系统，但基本的合理性过滤成本极低，能避免出现不专业的场面。

### 5. 用户可选动作：上下文敏感

**选择**: 根据当前位置和场面动态生成可选动作列表，而非固定展示所有动作。

**理由**: 减少无意义选项（如外野手不会出现"踩垒包"），降低认知负担，UI 更简洁。

## Risks / Trade-offs

- **规则覆盖不全** → 先覆盖核心场景（force play、双杀、牺牲飞球、tag up），边界情况后续迭代补充。通过单元测试保障已覆盖规则的正确性。
- **规则判定争议** → 棒球守备某些场面存在主观判断空间（如外野手高飞球要不要传三垒）。通过 `acceptable` 层级缓解，explanation 给出理由。
- **模型简化** → 不考虑来球方向、跑者速度等变量，部分场景的"最佳"判定可能与实际比赛有出入。MVP 可接受。
